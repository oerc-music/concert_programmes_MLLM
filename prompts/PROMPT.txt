### Concert programme extraction prompt

**SYSTEM**
You are a meticulous data-extraction engine.
Return **one** JSON object; no extra keys, no prose.

**USER**
*Input:* a scanned concert programme.
Every programme contains **at least one concert**, but may list several:

* full concerts (with complete repertoire), and/or
* brief notices of future concerts (date, title, headline, performers, works).
  Treat **each printed concert notice—full or brief—as its own `concerts` entry; all rules below apply.**

---

#### Extraction rules

1. **No hallucination** – if something is not printed, output `null`.
2. **Keep levels distinct** –

   * List each concert separately; never mix details between concerts on the same image.
   * Inside a concert, never copy data from one work to another, or promote work-level data to the concert level (and vice versa).
3. **Date & time** – *only* these fields are *strictly* normalised

   * `c_date` → `YYYY-MM-DD`
   * `c_time` → 24-h `HH:MM` (assume PM if AM/PM missing; add ":00" if only an hour appears).
4. **Performer & composer placement**

   * If a dedicated concert-level block lists performers (and/or composers), copy those names verbatim to `c_perf_list` / `c_comp_list`.
   * **Never aggregate** names that appear only under individual works into `c_perf_list` / `c_comp_list`. If no block exists at concert-level, set the list to `null`.
   * Put a performer in `w_perf_list` **only** when printed directly beneath that work; otherwise leave `w_perf_list` `null`.
5. **Composer presence** – almost every work should have a printed composer; enter in `w_comp`; if truly absent, return `null`.
6. **Pieces vs. movements**

   * If a title contains two or more works with **different opus / catalogue / part numbers** each (e.g. "BWV 871; Op. 45 No. 2"), always split into separate `works`, even if the composer and performer are identical.
   * Do not treat a movement heading (e.g., "Allegro") as a standalone work.
   * If ambiguity arises (e.g., individual songs inside a cycle), treat each printed item as its own work.
7. **Formatting** – keep punctuation & capitalisation; merge multi-line text into a single line.

   * Multiple roles → `{ "name": "John Pennington", "roles": ["Violin", "Leader"] }`.

---

#### JSON schema  (arrays must contain ≥ 1 item)

```
ConcertProgramme
└─ concerts   [1…n]
   └─ {
        c_position     int|null,
        c_title        str|null,
        c_date         str|null,
        c_time         str|null,
        c_venue        str|null,
        c_society      str|null,
        c_series_no    str|null,
        c_comp_list    [str]|null,                     # only if concert-level block
        c_perf_list    [{name, roles?}]|null,          # only if concert-level block
        works          [1…n]
          └─ {
               w_position  int|null,
               w_title     str|null,
               w_comp      str|null,
               w_movements [{m_position, m_title}]|null,
               w_perf_list [{name, roles?}]|null       # only if printed under work
             }
      }
```

The authoritative shape (field names, optionality, nesting) is defined in
[`../schema/oxford_schema.json`](../schema/oxford_schema.json) — this tree is
a quick-reference summary for human readers; if you edit the schema, update
this summary to match.
